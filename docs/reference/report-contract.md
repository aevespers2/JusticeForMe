# Report and Evidence Contract

## Current report schema

JusticeForMe emits JSON schema version `1.0` with this logical shape:

```json
{
  "schema_version": "1.0",
  "generated_at": "2026-07-20T21:00:00Z",
  "host": "example-host",
  "root_complete": true,
  "metrics": {
    "world_writable_etc": 0,
    "executable_files_etc": 0,
    "privileged_binaries": 0,
    "file_capabilities": 0,
    "package_integrity_findings": 0,
    "recent_etc_changes_30d": 0
  },
  "artifacts": ["etc-metadata.txt"],
  "notice": "Indicators require human review and are not proof of malicious activity."
}
```

## Field semantics

| Field | Current meaning | Important limitation |
|---|---|---|
| `schema_version` | Dashboard compatibility discriminator | Does not identify collector commit or evidence-envelope version |
| `generated_at` | UTC time generated by local Python runtime | Depends on local clock correctness |
| `host` | Short hostname observed locally | Not a durable or authenticated device identity |
| `root_complete` | Whether collector effective UID was root | Does not prove every check completed |
| `metrics` | Counts derived from generated artifacts and filesystem queries | Counts may combine expected state, findings, errors, or partial collection |
| `artifacts` | Expected artifact filenames | Does not bind filenames to hashes or prove presence |
| `notice` | Human-readable interpretation warning | Not a machine-enforced policy |

## Dashboard thresholds

The dashboard applies fixed triage thresholds:

| Metric | Warning threshold | High threshold |
|---|---:|---:|
| World-writable `/etc` entries | greater than 0 | 1 or more |
| Executable files in `/etc` | greater than 5 | 20 or more |
| Setuid/setgid binaries | greater than 20 | 60 or more |
| File capabilities | greater than 5 | 20 or more |
| Package-integrity findings | greater than 0 | 10 or more |
| Recent `/etc` changes | greater than 25 | 100 or more |

These values are review heuristics, not compliance rules, malware signatures, or host-specific approved baselines.

## Supporting artifacts

The collector currently names these artifacts:

- `etc-metadata.txt`
- `etc-sha256.txt`
- `etc-world-writable.txt`
- `etc-executable-files.txt`
- `privileged-binaries.txt`
- `file-capabilities.txt`
- `admin-access.txt`
- `persistence-and-loader.txt`
- `package-integrity.txt`
- `REPORT-SHA256SUMS.txt`

The final manifest hashes files in the report directory. It is an integrity aid, not a signature or remote attestation.

## Compatibility policy

- The existing dashboard accepts only report schema `1.0`.
- Additive optional fields may be proposed only if the current dashboard continues to render safely.
- A breaking field rename, type change, metric semantic change, or artifact semantic change requires a new schema version.
- Schema changes must include fixtures for valid, partial, malformed, unsupported, stale, and adversarial reports.
- A report-schema version must not be reused for materially different semantics.
- Transport, proposal, capability, and receipt envelopes are separate contracts and should not be overloaded into `report.json` without an accepted decision.

## Proposed future evidence envelope

A future Repository `0` adapter may wrap the report without modifying schema `1.0`:

```json
{
  "envelope_version": "justiceforme-evidence/0.1-proposal",
  "subject": {
    "device_id": "approved-device-identifier",
    "ownership_scope": "authorized-local-device"
  },
  "collector": {
    "repository": "aevespers2/JusticeForMe",
    "commit": "immutable-commit-sha",
    "script_sha256": "...",
    "effective_uid": 0
  },
  "execution": {
    "started_at": "...",
    "completed_at": "...",
    "platform": "linux",
    "status": "partial|complete|failed",
    "limitations": []
  },
  "report": {
    "schema_version": "1.0",
    "sha256": "..."
  },
  "manifest": {
    "sha256": "...",
    "artifact_count": 10
  }
}
```

This example is a design proposal only. It grants no collection, transport, capability, or remediation authority.

## Privacy and retention

Evidence may disclose usernames, group membership, filesystem paths, local services, package state, hostnames, and administrative configuration. Before transport or long-term retention, policy must define:

- purpose and authorized reviewers;
- data classification;
- redaction and minimization;
- encryption at rest and in transit;
- retention and deletion periods;
- incident and disclosure response;
- whether the report may leave the audited device;
- whether public issue, CI, artifact, or Pages storage is prohibited.

By default, raw audit evidence must not be committed to a public repository or uploaded to a public workflow artifact.
