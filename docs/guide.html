JusticeForMe documentation

Read-only Linux audit, local evidence, human review

JusticeForMe inventories selected Linux privilege and configuration indicators, preserves detailed local artifacts, and renders a compact summary in a static browser dashboard. It does not remediate systems, upload reports, determine attribution, or convert indicators into proof.

Safety boundary

Use only on systems you are authorized to inspect

Generated artifacts may contain hostnames, account names, permission relationships, package deviations, and configuration history. Keep report bundles private, preserve originals, and compare findings with trusted baselines and approved change records.

Project overview

A deliberately small evidence pipeline

01

Collect

A Bash script reads selected `/etc`, privilege, capability, administrative-access, persistence, and package-integrity state.

02

Preserve

The collector writes a timestamped local bundle with detailed artifacts, a compact JSON summary, and a SHA-256 manifest.

03

Review

The static dashboard reads a user-selected `report.json` locally and presents fixed heuristic priorities for human validation.

Architecture

Collection and interpretation remain separate

Collector boundary

  • Reads selected live-system state
  • Writes only the chosen output directory
  • May require `sudo` for completeness
  • Does not modify permissions or services
  • Does not transmit data

Dashboard boundary

  • Accepts schema version `1.0`
  • Reads a user-selected local JSON file
  • Displays six summary metrics
  • Uses fixed heuristic thresholds
  • Has no backend or report storage

Priority: one or more regular files or directories.

Executable `/etc` files

Review: above 5. Priority: 20 or more.

Privileged binaries

Review: above 20. Priority: 60 or more.

File capabilities

Review: above 5. Priority: 20 or more.

Package deviations

Review: one or more. Priority: 10 or more.

Recent `/etc` changes

Review: above 25. Priority: 100 or more.

These are display thresholds, not risk probabilities, compliance controls, or compromise verdicts. Platform-specific baselines remain essential.

Operations and recovery

Preserve first; analyze a copy

Collection runbook

  1. Confirm written authorization and scope.
  2. Record host, operator, UTC time source, and exact commit.
  3. Select a protected new output directory.
  4. Run the reviewed collector at the approved privilege level.
  5. Preserve terminal output and verify expected artifacts.
  6. Verify the checksum manifest from a trusted environment.
  7. Protect the original and analyze a copy.

Partial or interrupted collection

  1. Do not present the directory as complete evidence.
  2. Preserve it if the interruption itself matters.
  3. Record the failure point and environment.
  4. Correct capacity, permission, or tool issues.
  5. Run again to a new directory; never overwrite the first attempt.

Accidental public report publication

Treat exposure as a security incident. Remove the public artifact and Pages reference, preserve repository and deployment evidence, rotate any exposed secrets, assess disclosed identities and host details, notify the responsible owner through an approved channel, and record corrective actions. Deleting a commit alone may not remove cached or forked copies.
