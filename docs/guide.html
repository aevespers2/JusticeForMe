Read-only Linux audit architecture and integration guide

JusticeForMe collects a bounded local inventory of Linux privilege and configuration surfaces. It does not remediate the host, upload reports, or prove that an indicator is malicious.

Purpose and boundaries

The collector inventories selected /etc state, privileged binaries, Linux file capabilities, administrative access paths, persistence indicators, and package-integrity output. It writes a timestamped local evidence directory containing report.json, supporting artifacts, and a SHA-256 manifest.

The local dashboard accepts report schema 1.0 and applies fixed triage thresholds. Threshold labels are review aids—not compliance decisions, malware verdicts, attribution, or a comparison with a signed host baseline.

Architecture

  1. An authorized operator starts the collector on a Linux host.
  2. The collector reads supported local host surfaces and writes evidence into the selected output directory.
  3. Python serializes the compact report; the collector then creates a local SHA-256 manifest.
  4. The operator selects report.json in the browser dashboard. Parsing remains local.
  5. Any future Repository 0 integration must wrap the evidence in a versioned envelope before submitting a proposal to Repository 1.
Implemented: local read-only collection and local dashboard review. Proposed: device identity, evidence envelope, Repository 0 adapter, Repository 1 baseline comparison, capabilities, and remediation receipts.

Accessibility and review evidence

Status: DOCUMENTED_NOT_CERTIFIED. The repository defines a review protocol for the exact dashboard, guide, repository documentation, and rendered artifact rather than treating source validation as universal accessibility evidence.

  • Review keyboard access, focus order and visibility, labels, live status, and error recovery.
  • Review headings, landmarks, tables, metric semantics, and diagram prose with a screen reader.
  • Review 200% and 400% zoom, responsive reflow, long paths, hashes, code blocks, and narrow viewports.
  • Verify contrast, forced-colors behavior, reduced motion, and non-color severity communication.
  • Use synthetic fixtures and keep real host evidence out of public screenshots, recordings, logs, and artifacts.
  • Record the exact source, artifact digest, browser, viewport, input method, assistive technology, reviewer, limitations, and correction links.

A pass for one environment does not certify every environment. A moved head, changed renderer, discovered defect, or missing manual check requires a new, partial, failed, blocked, corrected, withdrawn, or superseded record.

Operations and evidence preservation

sudo ./audit/linux-privilege-audit.sh /protected/path/justiceforme-run
cd /protected/path/justiceforme-run
sha256sum -c REPORT-SHA256SUMS.txt
  • Use only on systems you own or are authorized to inspect.
  • Record the exact source commit, script hash, platform, and tool versions.
  • Preserve the original evidence directory before annotation or redaction.
  • Do not upload raw private audit evidence to public Pages, issues, pull requests, or public CI artifacts.
  • Freeze further automated changes when evidence identity, manifest integrity, or privileged access state is uncertain.

A.L.I.S.T.A.I.R.E. integration

JusticeForMe is a candidate Linux host-observation adapter for the portable first-install security foundation:

  1. Repository 0 supervises a local, approved observation task.
  2. JusticeForMe produces a report and evidence bundle.
  3. Repository 0 transforms evidence into a versioned proposal.
  4. Repository 1 validates device identity, schema, freshness, provenance, policy, and replay state.
  5. Only a separately approved, narrow capability may authorize reversible remediation.

The strongest current obstruction is that schema 1.0 has a hostname and global root flag but no authenticated device identity, per-check completion state, execution provenance, or replay protection. Integration must fail closed until those are supplied by an accepted envelope.

Developer workflow

Collector changes must remain read-only, bounded, privacy-aware, and honest about unsupported or partial collection. Dashboard changes must remain local-only, schema-aware, accessible, and safe when rendering untrusted report values.
